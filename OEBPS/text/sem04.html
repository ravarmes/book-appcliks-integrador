SEMANA 4 – Lógica Condicional e Modelagem de Dados

Nesta semana, você aprenderá a modelar o banco de dados do seu projeto, criando o Diagrama Entidade-Relacionamento (DER) e o Modelo Entidade-Relacionamento (MER).

A modelagem de dados é a etapa em que definimos quais informações o sistema vai armazenar e como elas se relacionam entre si. Um modelo bem construído facilita todo o desenvolvimento posterior e evita erros difíceis de corrigir depois.

Você Sabia?
  • O modelo Entidade-Relacionamento foi proposto por Peter Chen em 1976 e ainda é o padrão mais utilizado para modelagem conceitual de bancos de dados.
  • Um banco de dados bem modelado pode ser até 10x mais eficiente em consultas do que um banco não normalizado.
  • O Supabase (usado neste projeto) utiliza PostgreSQL, um dos bancos de dados relacionais mais robustos e usados no mundo.

Fonte: Chen, P. (1976). The Entity-Relationship Model.

4.1 Conceitos de Modelagem de Dados

Definição

Entidade: representa um objeto do mundo real sobre o qual queremos armazenar informações. Exemplos: Transação, Categoria. Em um banco de dados, cada entidade vira uma tabela.

Tabela 4.1 - Conceitos fundamentais de modelagem de dados.
Conceito Descrição Exemplo
Entidade Objeto ou conceito do mundo real Usuário, Categoria, Transação
Atributo Propriedade/característica de uma entidade nome, e-mail, data_nascimento
Chave Primária (PK) Atributo que identifica unicamente cada registro id (gerado automaticamente)
Chave Estrangeira (FK) Atributo que referencia a PK de outra tabela usuario_id em Transação
Relacionamento Associação entre duas entidades Usuário possui Transações

4.2 Cardinalidade dos Relacionamentos

A cardinalidade indica quantas ocorrências de uma entidade podem se associar a quantas ocorrências de outra entidade. É fundamental para modelar corretamente o banco.

Tabela 4.2 - Tipos de cardinalidade em relacionamentos.
Tipo Notação Exemplo
Um para Um (1:1) 1 — 1 Um País possui uma única Capital
Um para Muitos (1:N) 1 — N Um Usuário possui muitas Transações
Muitos para Muitos (N:M) N — M Um Produto pertence a muitas Categorias; uma Categoria tem muitos Produtos
Dica: Relacionamento N:M

Relacionamentos muitos-para-muitos (N:M) são implementados com uma tabela intermediária (tabela associativa). Por exemplo, a relação entre Produto e Pedido gera a tabela ItemPedido com as chaves estrangeiras de ambos.

4.3 Criando o DER do Projeto

O Diagrama Entidade-Relacionamento (DER) é a representação visual do modelo conceitual do banco de dados. Ele mostra as entidades, seus atributos e os relacionamentos entre elas.

Passo a Passo para criar o DER

  1. Liste todas as entidades identificadas nos seus requisitos funcionais.
  2. Para cada entidade, defina seus atributos (nome, tipo, obrigatório?).
  3. Identifique os relacionamentos entre as entidades.
  4. Defina a cardinalidade de cada relacionamento.
  5. Identifique as chaves primárias de cada entidade.
  6. Desenhe o diagrama usando uma ferramenta adequada.

Exemplo: DER de um App Financeiro

Para o app FinanCampus, as entidades e relacionamentos seriam:

  • Categoria (id, nome, icone, cor) — 1 categoria classifica N transações
  • Transação (id, tipo, descricao, valor, data, usuario_id FK, categoria_id FK) — cada registro é uma renda ou uma despesa
Onde ficam os dados do usuário?

O Supabase Auth gerencia automaticamente os dados de login (e-mail, senha, UID) na tabela interna auth.users. Essa tabela não é criada por você — ela já existe no Supabase. O campo usuario_id na tabela transacoes referencia o auth.users(id) para vincular cada transação ao usuário logado.

Por isso, não criamos uma tabela de usuários no nosso modelo — o Supabase cuida disso.
Por que usar "tipo" em vez de tabelas separadas?

Em vez de criar duas tabelas quase idênticas (despesas e receitas), usamos uma tabela única transacoes com o campo tipo ('Renda' ou 'Despesa'). Vantagens:

  • Menos tabelas para gerenciar e manter
  • CRUD único (mesma tela serve para renda e despesa)
  • Filtros simples: WHERE tipo = 'Despesa'
  • Alinhado com templates profissionais do FlutterFlow

Importante

Nomeie as entidades e atributos em letras minúsculas e sem acentos, usando underscores para separar palavras (snake_case). Por exemplo: data_criacao, usuario_id. Isso facilita a integração com o Supabase nas próximas semanas.

Por Que Separar os Dados em Tabelas? — Normalização

Um dos objetivos do DER é garantir que os dados não se repitam desnecessariamente no banco. Esse processo se chama normalização. Imagine uma tabela única que armazenasse, em cada linha de transação, o nome e o e-mail do usuário e o nome da categoria. Se a categoria "Alimentação" for renomeada para "Comida", você teria que atualizar centenas de linhas. Ao separá-la em uma tabela própria, basta alterar uma única linha.
