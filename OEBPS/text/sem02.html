SEMANA 2 – Navegação e Requisitos

Nesta semana, você irá definir a visão geral do sistema e levantar os requisitos funcionais e não funcionais do seu aplicativo. Um bom levantamento de requisitos é a base de todo o desenvolvimento.

Antes de construir qualquer funcionalidade, é necessário saber o que o sistema deve fazer e como deve se comportar. Esse processo — chamado de engenharia de requisitos — evita retrabalho e garante que o produto final atenda às reais necessidades do usuário.

Você Sabia?
  • Segundo o Standish Group Chaos Report, cerca de 30% dos projetos de software falham por requisitos mal definidos ou incompletos.
  • Um requisito bem definido precisa ser: claro, verificável, viável e necessário.
  • O levantamento de requisitos é uma atividade contínua: mesmo durante o desenvolvimento, requisitos podem ser refinados.

Fonte: Standish Group – Chaos Report (2020). Sommerville, I. – Engenharia de Software, 10ª ed.

2.1 Visão Geral do Sistema

A visão do sistema é um documento breve que resume o propósito do app, seus principais usuários e as funcionalidades centrais. Ela serving como "bússola" para toda a equipe durante o projeto.

Estrutura da Visão do Sistema

A visão pode ser expressa numa única frase usando o modelo a seguir:

Template: Frase de Visão

Para [público-alvo], que [necessidade ou problema], o [nome do app] é um [categoria de produto] que [benefício-chave]. Diferentemente de [alternativa atual], nosso produto [diferencial].

Exemplo

Para estudantes universitários que têm dificuldade em controlar os gastos mensais, o FinanCampus é um aplicativo mobile que registra transações financeiras (rendas e despesas) de forma simples e visual. Diferentemente de planilhas e aplicativos complexos, nosso produto oferece categorização automática e alertas de orçamento com interface intuitiva.

2.2 Requisitos Funcionais

Os requisitos funcionais (RF) descrevem o que o sistema deve fazer — as funcionalidades visíveis ao usuário. São as ações que o app precisa realizar para cumprir seu propósito.

Definição

Requisito Funcional (RF): descreve uma função ou comportamento que o sistema deve executar. Define o que o sistema faz em resposta a entradas ou condições específicas.

Como Escrever um Requisito Funcional

Use sempre o formato: "O sistema deve [verbo] [complemento]."

Tabela 2.1 - Exemplos de requisitos funcionais para um app financeiro.
ID Requisito Funcional Prioridade
RF01 O sistema deve permitir o cadastro de novos usuários com nome, e-mail e senha. Alta
RF02 O sistema deve permitir o registro de uma transação com tipo (Renda/Despesa), valor, categoria e data. Alta
RF03 O sistema deve exibir um gráfico de transações por categoria no mês atual. Média
RF04 O sistema deve enviar uma notificação quando o limite de orçamento for atingido. Baixa
RF05 O sistema deve permitir exportar o extrato mensal em formato PDF. Baixa
Dica

Use prioridades Alta, Média e Baixa para classificar os requisitos. Foque primeiro nos de alta prioridade — eles formam o chamado Produto Mínimo Viável (MVP).

User Stories: A Voz do Usuário

Uma técnica muito utilizada em métodos ágeis para capturar requisitos é a User Story (História de Usuário). Diferente do formato técnico tradicional, ela escreve o requisito do ponto de vista do usuário, deixando claro quem precisa, o que precisa e por quê:

"Como [tipo de usuário], quero [ação], para [objetivo/benefício]."

Cada User Story pode originar um ou mais Requisitos Funcionais formais. A Figura 2.2 mostra como uma User Story se transforma no RF documentado.

2.3 Requisitos Não Funcionais

Os requisitos não funcionais (RNF) descrevem como o sistema deve se comportar — qualidade, desempenho, segurança, usabilidade. Eles não estão diretamente ligados a uma funcionalidade específica, mas impactam toda a experiência de uso. A Figura 2.1 ilustra a diferença entre os dois tipos de requisitos.

Definição

Requisito Não Funcional (RNF): especifica critérios de qualidade do sistema como um todo: desempenho, segurança, usabilidade, confiabilidade, compatibilidade e manutenibilidade.
Tabela 2.2 - Categorias comuns de requisitos não funcionais.
Categoria Exemplo de RNF
Desempenho O app deve carregar a tela principal em menos de 2 segundos.
Segurança As senhas dos usuários devem ser armazenadas com criptografia (hash).
Usabilidade Um usuário novo deve conseguir cadastrar sua primeira transação em menos de 1 minuto.
Compatibilidade O app deve funcionar em dispositivos Android com versão 8.0 ou superior.
Disponibilidade O sistema deve estar disponível 99% do tempo (banco de dados Firebase).
Importante

Nunca confunda o que com o como. Um requisito funcional descreve uma funcionalidade ("o usuário pode cadastrar transações"). Um requisito não funcional define a qualidade desse comportamento ("o cadastro deve ser concluído em menos de 3 segundos").

2.5 Estudo de Caso: FinanCampus

Veja como o grupo do FinanCampus definiu a visão e os requisitos do app.

📱 Estudo de Caso — Semana 2

Frase de Visão: Para estudantes universitários que perdem o controle dos gastos mensais, o FinanCampus é um aplicativo mobile que registra e categoriza transações financeiras de forma simples e visual. Diferentemente de planilhas e apps bancários complexos, nosso produto oferece visão rápida do saldo com alertas de orçamento.

Requisitos Funcionais (RF) — Alta Prioridade:

  • RF01 — O sistema deve permitir cadastro e login por e-mail e senha.
  • RF02 — O sistema deve permitir registrar uma transação (tipo, valor, categoria, data, descrição).
  • RF03 — O sistema deve listar todas as transações do mês atual do usuário logado.
  • RF04 — O sistema deve exibir o saldo do mês (rendas − despesas).
  • RF05 — Gráfico de transações por categoria.

Requisitos Funcionais (RF) — Média/Baixa Prioridade:

  • RF06 — Definir orçamento mensal por categoria.
  • RF07 — Notificação ao atingir 80% do orçamento.
  • RF08 — Exportar extrato mensal em PDF.

Requisitos Não Funcionais (RNF):

  • RNF01 — Desempenho: tela principal deve carregar em menos de 2 segundos.
  • RNF02 — Segurança: senhas armazenadas com hash (Firebase Auth — bcrypt).
  • RNF03 — Compatibilidade: Android 8.0 ou superior.
  • RNF04 — Usabilidade: novo usuário registra primeira transação em menos de 1 minuto.
