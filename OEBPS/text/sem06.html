SEMANA 6 – CRUD Completo

Nesta semana, você irá configurar o banco de dados no Supabase, conectá-lo ao FlutterFlow e implementar os fluxos de CRUD do seu projeto, além de preencher os dados iniciais (seeds) para permitir testes realistas.

O acrônimo CRUD representa as quatro operações básicas de qualquer sistema que gerencia dados: Create (criar), Read (ler), Update (atualizar) e Delete (excluir). No FlutterFlow, essas operações são configuradas via Actions e conectadas ao banco de dados.

Você Sabia?
  • O conceito de CRUD foi popularizado por James Martin em 1983 no livro Managing the Data-base Environment.
  • Praticamente todo app de mercado (redes sociais, e-commerce, bancos) é fundamentalmente construído sobre operações CRUD.
  • No Supabase, cada operação CRUD é uma chamada a uma API RESTful gerada automaticamente a partir das suas tabelas.

Fonte: Martin, J. (1983). Managing the Data-base Environment.

6.1 Configurando o Supabase no FlutterFlow

Antes de implementar qualquer fluxo de CRUD, o banco de dados precisa estar criado e conectado ao FlutterFlow. O Supabase é a plataforma de backend utilizada neste projeto — ela fornece um banco de dados PostgreSQL na nuvem, acessível direto pelo FlutterFlow sem necessidade de código.

Por que Supabase e não banco local?

O FlutterFlow não possui suporte a banco de dados local relacional (como SQLite) sem código personalizado avançado. As únicas opções nativas para persistência de dados são serviços na nuvem: Supabase, Firebase ou uma API REST. Para este projeto, o Supabase é a escolha ideal — é gratuito, visual, relacional (SQL) e integrado nativamente ao FlutterFlow.

Passo 1 — Criar o Projeto no Supabase

  1. Acesse supabase.com e crie uma conta (pode usar o Google).
  2. Clique em New Project.
  3. Defina um nome para o projeto (ex.: financampus), escolha uma senha forte para o banco e selecione a região South America (São Paulo).
  4. Aguarde a criação do projeto (cerca de 2 minutos).
  5. Anote as credenciais: Project URL e API Key (anon) — você vai precisar delas no passo 3.
Segurança das Chaves

A anon key é pública e pode ser usada no frontend (FlutterFlow). A service role key é privada e nunca deve ser exposta no código do app — ela bypassa todas as regras de segurança do banco.

Passo 2 — Criar as Tabelas no Supabase

Na Semana 5 você preparou o script SQL das tabelas a partir do brModelo. Agora é só executá-lo diretamente no Supabase:

  1. No painel do Supabase, acesse SQL Editor (menu lateral).
  2. Clique em New query.
  3. Cole o script SQL exportado pelo brModelo na Semana 5.
  4. Clique em Run (▶). Todas as tabelas e relacionamentos serão criados de uma só vez.
  5. Confirme em Table Editor que as tabelas aparecem listadas.
RLS será configurado na Semana 9

Nesta semana, deixe o RLS desabilitado para que você consiga testar as operações CRUD livremente durante o desenvolvimento. Na Semana 9, você ativará o RLS e configurará as políticas de segurança, garantindo que cada usuário acesse apenas seus próprios dados.

Passo 3 — Conectar o Supabase ao FlutterFlow

  1. No FlutterFlow, clique no ícone de engrenagem ⚙️ na parte inferior do menu lateral para abrir o App Settings.
  2. No painel esquerdo, role até a seção Integrations e clique em Supabase.
  3. Ative o toggle Enable Supabase — os campos de credenciais serão exibidos.
  4. Preencha os campos:
    • API URL: cole a Project URL copiada do Supabase no Passo 1.
    • Anon Key: cole a API Key (anon) copiada do Supabase no Passo 1.
  5. Clique em Get Schema — o FlutterFlow importará automaticamente todas as tabelas e colunas criadas no Passo 2. A partir deste momento, as Actions de banco de dados (Insert Row, Update Row, Delete Row, Query Rows) estarão disponíveis para uso em qualquer botão ou evento do app.

6.2 Autenticação com Supabase

Antes de implementar o CRUD, vamos configurar a autenticação do app. Isso é fundamental: ao criar registros no banco, o campo usuario_id será preenchido automaticamente com o auth.uid() do usuário logado — garantindo que cada pessoa acesse apenas seus próprios dados.

Passo 1 — Habilitar a Autenticação no Supabase

  1. No painel do Supabase, acesse Authentication → Providers.
  2. Habilite o provedor Email (e-mail e senha).
  3. Desative a opção Confirm email para facilitar os testes (em produção, mantenha ativada).

Passo 2 — Ativar a Autenticação no FlutterFlow

  1. No FlutterFlow, acesse Settings → Supabase.
  2. Ative a opção Use Supabase Auth.
  3. Configure a Entry Page (tela de Login) e a Logged In Page (tela Home). O FlutterFlow redirecionará automaticamente o usuário para a tela correta ao abrir o app.

Passo 3 — Implementar as Telas de Autenticação

As telas de Login e Cadastro usam as Actions de autenticação do Supabase, listadas na tabela abaixo:

Tabela 6.1 - Actions de autenticação Supabase no FlutterFlow.
Action Descrição Onde usar
Sign Up (Email) Cria um novo usuário com e-mail e senha Botão "Criar Conta"
Sign In (Email) Autentica um usuário existente Botão "Entrar"
Sign Out Encerra a sessão do usuário Botão "Sair" (Configurações)

Implementando o Cadastro (Sign Up)

  1. Na tela de Cadastro, adicione campos para E-mail, Senha e Confirmar Senha.
  2. No botão "Criar Conta", configure a Action: Supabase Auth → Sign Up (Email).
  3. Mapeie os campos de e-mail e senha aos respectivos TextFields.
  4. Adicione uma Action subsequente: Navigate To → Home.

Implementando o Login (Sign In)

  1. Na tela de Login, adicione campos para E-mail e Senha.
  2. No botão "Entrar", configure a Action: Supabase Auth → Sign In (Email).
  3. Mapeie os campos e adicione: Navigate To → Home.

Implementando o Logout (Sign Out)

  1. Na tela de Configurações (ou Perfil), adicione um botão "Sair".
  2. Configure a Action: Supabase Auth → Sign Out.
  3. Adicione: Navigate To → Login (com Replace Route ativado para evitar que o botão "Voltar" retorne à tela anterior).
Como funciona o JWT

Ao fazer login, o Supabase retorna um JWT (JSON Web Token) — um token criptografado que identifica o usuário. O FlutterFlow armazena automaticamente esse token e o envia em cada requisição ao banco. O Supabase usa o auth.uid() nas políticas RLS para filtrar os dados. Por isso é fundamental implementar o login antes de testar o CRUD com dados pessoais.

Tabela 6.2 - Correspondência entre operações CRUD e SQL/API.
Operação SQL HTTP (API REST) FlutterFlow Action

Read SELECT FROM GET Query Rows / Get Single Row

Definição

Action (FlutterFlow): bloco de lógica executado em resposta a um evento (clique de botão, carregamento de página, etc.). As Actions conectam a interface do usuário com o banco de dados e outras integrações.

6.4 Implementando os Fluxos CRUD

Com o banco criado e o schema carregado, é hora de implementar os fluxos de interação com os dados. O CRUD (Create, Read, Update, Delete) representa as quatro operações fundamentais sobre qualquer tabela. No FlutterFlow, cada operação corresponde a uma Action específica que se comunica com o Supabase via API REST, como ilustrado na Figura 6.1.

Nesta semana você implementa os fluxos de escrita de dados: criar um novo registro (Create), editá-lo (Update) e excluí-lo (Delete). Ao longo desses fluxos você usará Actions de navegação — Navigate Back e Navigate To — apenas para redirecionar o usuário depois que uma operação é concluída (ex.: voltar para a tela anterior após salvar). Isso é diferente de arquitetar a navegação do app: a estrutura completa de telas, o diagrama de fluxo e as listas dinâmicas conectadas ao banco são o foco da Semana 7.

Antes de detalhar cada fluxo, é fundamental entender a diferença entre dois conceitos essenciais do FlutterFlow: widgets e propriedades.

Widgets vs. Propriedades

Widget é um elemento visual que você arrasta para a tela: Button, TextField, ListView, Column, Row, Container, Card, Icon, Text etc. Cada widget aparece na árvore de componentes no painel esquerdo do FlutterFlow.

Propriedade é uma configuração de um widget já inserido na tela, ajustada no painel direito: por exemplo, o label de um Button, a initialValue de um TextField, o mainAxisAlignment de uma Column, ou a font size de um Text. Propriedades não são elementos visuais independentes — elas configuram a aparência e o comportamento de um widget existente.

🟢 Fluxo de Create (Inserção)

O fluxo de Create envolve uma tela de formulário onde o usuário preenche os dados e os salva no banco de dados via Supabase.

  1. Crie a tela de formulário.
    No painel de widgets, localize a seção Commonly Used Elements e arraste um widget Column como contêiner principal da tela — ele empilha os elementos verticalmente.
  2. Adicione os campos de entrada.
    Para cada dado a ser coletado, arraste um widget TextField (encontrado em Base Elements) dentro da Column. Configure as seguintes propriedades de cada TextField no painel direito:
    • Label Text — rótulo exibido dentro do campo (ex.: "Descrição").
    • Controller Path — vincula o campo a uma variável de estado para capturar o valor digitado.
    • Keyboard Type — define o tipo de teclado (texto, número, e-mail, data etc.).
    • Validator — regra de validação (ex.: campo obrigatório, valor mínimo).
    Para campos de seleção (ex.: categoria), use o widget DropDown em vez do TextField.
    3. Dropdown vinculado à tabela real

    Como o banco já está conectado via Get Schema, o DropDown de categoria pode ser populado diretamente com os dados da tabela categorias do Supabase — sem valores estáticos. Para isso:

    1. Selecione o widget DropDown na árvore de componentes.
    2. No painel direito, em Options Source, escolha From Database.
    3. Configure uma Query Rows na tabela categorias (sem filtros, para listar todas as categorias disponíveis).
    4. Mapeie o campo nome como label (o texto exibido ao usuário) e o campo id como value (o valor armazenado no banco como categoria_id na transação).

    Dessa forma, quando o usuário selecionar uma categoria, o id real da categoria será salvo na tabela transacoes via a action Insert Row, mantendo a integridade referencial da Foreign Key.

  3. Envolva os campos em um widget Form Validation.
    Encontrado na seção de widgets avançados (pesquise por "Form Validation" com Ctrl+F), ele agrupa os TextFields e permite validar todos de uma vez antes de salvar.
  4. Adicione o botão "Salvar".
    Arraste um widget Button (em Commonly Used Elements) abaixo dos campos. Configure as propriedades:
    • Text — defina o rótulo do botão: "Salvar".
    • Width — defina como Infinity (largura total) para um visual mais moderno.
  5. Configure a Action do botão.
    Clique no botão e abra o painel de Actions. Adicione as seguintes ações na ordem:
    1. Validate Form — valida todos os campos antes de prosseguir.
    2. Supabase → Insert Row — selecione a tabela correta e mapeie cada coluna ao valor do TextField correspondente.
    3. Navigate Back (ou Navigate To) — redireciona para a tela de listagem após o salvamento.

🟡 Fluxo de Update (Atualização)

O fluxo de Update reaproveita a estrutura do formulário de Create, mas pré-preenche os campos com os dados do registro existente e usa a action Update Row.

  1. Crie (ou duplique) a tela de edição.
    Você pode duplicar a tela de criação para reaproveitar os widgets já configurados. Adicione um Page Parameter do tipo int (ou uuid) chamado transacaoId para receber o ID do registro.
  2. Carregue os dados existentes.
    Configure um Backend Query do tipo Get Single Row na tela, filtrando por id = transacaoId. Isso busca o registro ao abrir a tela.
  3. Pré-preencha os campos.
    Para cada widget TextField, configure a propriedade Initial Value como o campo correspondente do resultado da query (ex.: o initialValue do campo "Descrição" recebe o valor de transacao.descricao).
  4. Configure a Action do botão "Salvar".
    1. Validate Form — valida os campos.
    2. Supabase → Update Row — selecione a tabela, defina o filtro id = transacaoId e mapeie cada coluna ao novo valor do campo.
    3. Navigate Back — retorna para a tela de detalhe.

🔴 Fluxo de Delete (Exclusão)

  1. Adicione o botão "Excluir".
    Na tela de detalhe, arraste um widget Button. Configure a propriedade Text como "Excluir" e opcionalmente use um widget Icon (ícone de lixeira) à esquerda para reforço visual.
  2. Configure um diálogo de confirmação.
    Na Action do botão, adicione primeiro: Alert Dialog → mensagem "Deseja excluir este registro?" com os botões "Cancelar" e "Confirmar". A exclusão só ocorre na ramificação de confirmação.
  3. Configure a Action de exclusão.
    Dentro da ramificação "Confirmar" do diálogo, adicione:
    1. Supabase → Delete Row → filtre por id = transacaoId.
    2. Navigate Back → retorna para a listagem após a exclusão.
Importante

Sempre valide os dados antes de executar um Insert ou Update. No FlutterFlow, use a Action Validate Form antes da chamada ao banco. Para exclusões, sempre exiba uma confirmação ao usuário — operações de delete geralmente são irreversíveis.

6.5 Dados Iniciais (Seeds)

Os seeds (ou dados de semente) são registros iniciais inseridos no banco de dados para popular as tabelas com informações de exemplo. Eles são essenciais para testar o funcionamento do app durante o desenvolvimento.

Como Criar Seeds no Supabase

No painel do Supabase, acesse Table Editor → [nome da tabela] → Insert Row para adicionar registros manualmente. Alternativamente, use o editor SQL:

Dica

Crie pelo menos 5 a 10 registros de seed para cada tabela auxiliar (categorias, tags, tipos, etc.). Isso permite testar listagens, filtros e relacionamentos de forma realista durante o desenvolvimento.
