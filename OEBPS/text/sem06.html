SEMANA 6 – CRUD Completo

Nesta semana, você irá configurar o Firebase no FlutterFlow, criar as coleções no Firestore e implementar os fluxos de CRUD do seu projeto, além de preencher os dados iniciais (seeds) para permitir testes realistas.

O acrônimo CRUD representa as quatro operações básicas de qualquer sistema que gerencia dados: Create (criar), Read (ler), Update (atualizar) e Delete (excluir). No FlutterFlow, essas operações são configuradas via Actions e conectadas ao banco de dados.

Você Sabia?
  • O conceito de CRUD foi popularizado por James Martin em 1983 no livro Managing the Data-base Environment.
  • Praticamente todo app de mercado (redes sociais, e-commerce, bancos) é fundamentalmente construído sobre operações CRUD.
  • No Firebase/Firestore, cada operação CRUD é uma chamada direta à SDK do Firebase, integrada nativamente ao FlutterFlow — sem necessidade de APIs intermediárias.

Fonte: Martin, J. (1983). Managing the Data-base Environment.

6.1 Configurando o Firebase no FlutterFlow

Antes de implementar qualquer fluxo de CRUD, o banco de dados precisa estar criado e conectado ao FlutterFlow. O Firebase é a plataforma de backend utilizada neste projeto — ela fornece o Firestore (banco de dados NoSQL) na nuvem, acessível direto pelo FlutterFlow sem necessidade de código.

Por que Firebase e não banco local?

O FlutterFlow não possui suporte a banco de dados local relacional (como SQLite) sem código personalizado avançado. As únicas opções nativas para persistência de dados são serviços na nuvem: Firebase, Supabase ou uma API REST. Para este projeto, o Firebase é a escolha ideal — é gratuito (plano Spark), visual, integrado nativamente ao FlutterFlow e não exige conhecimento de SQL.

Passo 1 — Criar o Projeto no Firebase

  1. Acesse console.firebase.google.com e faça login com sua conta Google.
  2. Clique em Adicionar projeto (ou Add project).
  3. Defina um nome para o projeto (ex.: financampus).
  4. Desative o Google Analytics (opcional para projetos acadêmicos) e clique em Criar projeto.
  5. Aguarde a criação do projeto (cerca de 1 minuto).

Passo 2 — Conectar o Firebase ao FlutterFlow

O FlutterFlow oferece uma conexão automática com o Firebase. Siga os passos:

  1. No FlutterFlow, clique no ícone de engrenagem ⚙️ na parte inferior do menu lateral para abrir o App Settings.
  2. No painel esquerdo, role até a seção Integrations e clique em Firebase.
  3. Clique em Connect Firebase Project — o FlutterFlow solicitará acesso à sua conta Google para listar os projetos Firebase disponíveis.
  4. Selecione o projeto criado no Passo 1 e confirme a conexão.
  5. O FlutterFlow configurará automaticamente as credenciais (API Key, Project ID, etc.).
Segurança das Credenciais

As credenciais do Firebase são configuradas automaticamente pelo FlutterFlow. Nunca compartilhe o arquivo google-services.json (Android) ou GoogleService-Info.plist (iOS) publicamente — eles contêm as chaves de acesso ao seu projeto Firebase.

Passo 3 — Criar as Coleções no Firestore

Com o Firebase conectado, agora vamos criar as coleções diretamente no FlutterFlow:

  1. No FlutterFlow, acesse Firestore no menu lateral (ícone de banco de dados).
  2. Clique em + (Create Collection).
  3. Nomeie a coleção (ex.: categorias) e adicione os campos definidos na Semana 5:
    • nome — String
    • icone — String
    • cor — String
  4. Repita para a coleção transacoes com os campos:
    • tipo — String
    • descricao — String
    • valor — Double
    • data — Timestamp
    • usuario_uid — String
    • categoria_ref — Document Reference (→ categorias)
    • created_at — Timestamp
  5. O FlutterFlow criará automaticamente as coleções no Firestore e disponibilizará as Actions de CRUD (Create Document, Update Document, Delete Document, Query Collection).
Security Rules serão configuradas na Semana 9

Nesta semana, as regras de segurança do Firestore estarão no modo teste (qualquer pessoa autenticada pode ler/escrever) para que você consiga testar as operações CRUD livremente durante o desenvolvimento. Na Semana 9, você configurará as Security Rules, garantindo que cada usuário acesse apenas seus próprios dados.

6.2 Autenticação com Firebase

Antes de implementar o CRUD, vamos configurar a autenticação do app. Isso é fundamental: ao criar documentos no banco, o campo usuario_uid será preenchido automaticamente com o UID do usuário logado — garantindo que cada pessoa acesse apenas seus próprios dados.

Passo 1 — Habilitar a Autenticação no Firebase

  1. No console do Firebase, acesse Authentication no menu lateral.
  2. Clique em Get Started (ou Primeiros passos).
  3. Na aba Sign-in method, habilite o provedor Email/Password (e-mail e senha).
  4. Salve as configurações.

Passo 2 — Ativar a Autenticação no FlutterFlow

  1. No FlutterFlow, acesse Settings → Firebase.
  2. Ative a opção Authentication.
  3. Configure a Entry Page (tela de Login) e a Logged In Page (tela Home). O FlutterFlow redirecionará automaticamente o usuário para a tela correta ao abrir o app.

Passo 3 — Implementar as Telas de Autenticação

As telas de Login e Cadastro usam as Actions de autenticação do Firebase, listadas na tabela abaixo:

Tabela 6.1 - Actions de autenticação Firebase no FlutterFlow.

Implementando o Cadastro (Create Account)

  1. Na tela de Cadastro, adicione campos para E-mail, Senha e Confirmar Senha.
  2. No botão "Criar Conta", configure a Action: Auth → Create Account (Email).
  3. Mapeie os campos de e-mail e senha aos respectivos TextFields.
  4. Adicione uma Action subsequente: Navigate To → Home.

Implementando o Login (Log In)

  1. Na tela de Login, adicione campos para E-mail e Senha.
  2. No botão "Entrar", configure a Action: Auth → Log In (Email).
  3. Mapeie os campos e adicione: Navigate To → Home.

Implementando o Logout (Log Out)

  1. Na tela de Configurações (ou Perfil), adicione um botão "Sair".
  2. Configure a Action: Auth → Log Out.
  3. Adicione: Navigate To → Login (com Replace Route ativado para evitar que o botão "Voltar" retorne à tela anterior).
Como funciona a autenticação Firebase

Ao fazer login, o Firebase retorna um token de autenticação que identifica o usuário. O FlutterFlow armazena automaticamente esse token e o envia em cada requisição ao Firestore. As Security Rules usam o request.auth.uid para filtrar os dados. Por isso é fundamental implementar o login antes de testar o CRUD com dados pessoais.

Definição

6.4 Implementando os Fluxos CRUD

Com as coleções criadas e o Firebase conectado, é hora de implementar os fluxos de interação com os dados. O CRUD (Create, Read, Update, Delete) representa as quatro operações fundamentais sobre qualquer coleção. No FlutterFlow, cada operação corresponde a uma Action específica que se comunica com o Firestore via a SDK do Firebase, como ilustrado na Figura 6.1.

Nesta semana você implementa os fluxos de escrita de dados: criar um novo documento (Create), editá-lo (Update) e excluí-lo (Delete). Ao longo desses fluxos você usará Actions de navegação — Navigate Back e Navigate To — apenas para redirecionar o usuário depois que uma operação é concluída (ex.: voltar para a tela anterior após salvar). Isso é diferente de arquitetar a navegação do app: a estrutura completa de telas, o diagrama de fluxo e as listas dinâmicas conectadas ao banco são o foco da Semana 7.

Antes de detalhar cada fluxo, é fundamental entender a diferença entre dois conceitos essenciais do FlutterFlow: widgets e propriedades.

Widgets vs. Propriedades

Widget é um elemento visual que você arrasta para a tela: Button, TextField, ListView, Column, Row, Container, Card, Icon, Text etc. Cada widget aparece na árvore de componentes no painel esquerdo do FlutterFlow.

Propriedade é uma configuração de um widget já inserido na tela, ajustada no painel direito: por exemplo, o label de um Button, a initialValue de um TextField, o mainAxisAlignment de uma Column, ou a font size de um Text. Propriedades não são elementos visuais independentes — elas configuram a aparência e o comportamento de um widget existente.

🟢 Fluxo de Create (Inserção)

O fluxo de Create envolve uma tela de formulário onde o usuário preenche os dados e os salva no Firestore.

  1. Crie a tela de formulário.
    No painel de widgets, localize a seção Commonly Used Elements e arraste um widget Column como contêiner principal da tela — ele empilha os elementos verticalmente.
  2. Adicione os campos de entrada.
    Para cada dado a ser coletado, arraste um widget TextField (encontrado em Base Elements) dentro da Column. Configure as seguintes propriedades de cada TextField no painel direito:
    • Label Text — rótulo exibido dentro do campo (ex.: "Descrição").
    • Controller Path — vincula o campo a uma variável de estado para capturar o valor digitado.
    • Keyboard Type — define o tipo de teclado (texto, número, e-mail, data etc.).
    • Validator — regra de validação (ex.: campo obrigatório, valor mínimo).
    Para campos de seleção (ex.: categoria), use o widget DropDown em vez do TextField.
    3. Dropdown vinculado à coleção real

    Como as coleções já estão criadas no Firestore, o DropDown de categoria pode ser populado diretamente com os dados da coleção categorias — sem valores estáticos. Para isso:

    1. Selecione o widget DropDown na árvore de componentes.
    2. No painel direito, em Options Source, escolha Firestore Query.
    3. Configure uma Query Collection na coleção categorias (sem filtros, para listar todas as categorias disponíveis).
    4. Mapeie o campo nome como label (o texto exibido ao usuário) e o Document Reference como value (o valor armazenado como categoria_ref na transação).

    Dessa forma, quando o usuário selecionar uma categoria, a referência do documento será salva na coleção transacoes via a action Create Document, mantendo o relacionamento entre as coleções.

  3. Envolva os campos em um widget Form Validation.
    Encontrado na seção de widgets avançados (pesquise por "Form Validation" com Ctrl+F), ele agrupa os TextFields e permite validar todos de uma vez antes de salvar.
  4. Adicione o botão "Salvar".
    Arraste um widget Button (em Commonly Used Elements) abaixo dos campos. Configure as propriedades:
    • Text — defina o rótulo do botão: "Salvar".
    • Width — defina como Infinity (largura total) para um visual mais moderno.
  5. Configure a Action do botão.
    Clique no botão e abra o painel de Actions. Adicione as seguintes ações na ordem:
    1. Validate Form — valida todos os campos antes de prosseguir.
    2. Firestore → Create Document — selecione a coleção correta e mapeie cada campo ao valor do TextField correspondente. O campo usuario_uid deve ser preenchido com o Authenticated User UID.
    3. Navigate Back (ou Navigate To) — redireciona para a tela de listagem após o salvamento.
E o Read (Leitura)?

O fluxo de Read — construção das telas de listagem com ListView dinâmico conectado ao Firestore — é desenvolvido na Semana 7, junto com o mapeamento completo das telas e o diagrama de navegação do app. Aqui na Semana 6 o foco é nos formulários de criação, edição e exclusão.

🟡 Fluxo de Update (Atualização)

O fluxo de Update reaproveita a estrutura do formulário de Create, mas pré-preenche os campos com os dados do documento existente e usa a action Update Document.

  1. Crie (ou duplique) a tela de edição.
    Você pode duplicar a tela de criação para reaproveitar os widgets já configurados. Adicione um Page Parameter do tipo Document Reference chamado transacaoRef para receber a referência do documento.
  2. Carregue os dados existentes.
    Configure um Backend Query do tipo Get Document na tela, usando o transacaoRef como referência. Isso busca o documento ao abrir a tela.
  3. Pré-preencha os campos.
    Para cada widget TextField, configure a propriedade Initial Value como o campo correspondente do resultado da query (ex.: o initialValue do campo "Descrição" recebe o valor de transacao.descricao).
  4. Configure a Action do botão "Salvar".
    1. Validate Form — valida os campos.
    2. Firestore → Update Document — use o transacaoRef para atualizar o documento correto e mapeie cada campo ao novo valor.
    3. Navigate Back — retorna para a tela de detalhe.

🔴 Fluxo de Delete (Exclusão)

  1. Adicione o botão "Excluir".
    Na tela de detalhe, arraste um widget Button. Configure a propriedade Text como "Excluir" e opcionalmente use um widget Icon (ícone de lixeira) à esquerda para reforço visual.
  2. Configure um diálogo de confirmação.
    Na Action do botão, adicione primeiro: Alert Dialog → mensagem "Deseja excluir este registro?" com os botões "Cancelar" e "Confirmar". A exclusão só ocorre na ramificação de confirmação.
  3. Configure a Action de exclusão.
    Dentro da ramificação "Confirmar" do diálogo, adicione:
    1. Firestore → Delete Document → use o transacaoRef.
    2. Navigate Back → retorna para a listagem após a exclusão.
Importante

Sempre valide os dados antes de executar um Create Document ou Update Document. No FlutterFlow, use a Action Validate Form antes da chamada ao banco. Para exclusões, sempre exiba uma confirmação ao usuário — operações de delete geralmente são irreversíveis.

6.5 Dados Iniciais (Seeds)

Os seeds (ou dados de semente) são registros iniciais inseridos no banco de dados para popular as coleções com informações de exemplo. Eles são essenciais para testar o funcionamento do app durante o desenvolvimento.

Como Criar Seeds no Firestore

  • Delete: Link "Excluir" (em vermelho) na tela "Editar Transação" → Alert Dialog de confirmação → Delete Document → Navigate Back para "Início".
