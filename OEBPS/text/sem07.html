SEMANA 7 – Listas Dinâmicas e Mapeamento de Telas

Nesta semana, você irá mapear todas as telas do seu aplicativo, construir o diagrama de navegação completo e implementar as listas dinâmicas que exibem dados do banco em tempo real — conectando tudo que foi criado nas semanas anteriores.

Até agora, nossa navegação foi apenas pontual. Na Semana 6, você usou Navigate Back e Navigate To dentro dos fluxos de Create, Update e Delete para redirecionar o usuário após uma operação rápida. Isso atende a funções isoladas, mas não define a estrutura geral do app.

Nesta Semana 7, vamos elevar o padrão: você vai estruturar a Arquitetura de Navegação do app como um todo. Precisamos definir todas as telas, organizar os caminhos globais (como a Bottom Navigation Bar e fluxos de login) e desenhar um diagrama que será o roteiro do seu projeto. Com esse mapa pronto, passaremos a exibir dados reais do banco construindo listas dinâmicas através de ListView e Backend Query.

Um app com boa experiência de usuário precisa ter uma navegação clara e previsível. Mapear todas as telas antes de implementar evita problemas de fluxo e garante que o usuário sempre saiba onde está e como chegar onde quer.

Você Sabia?
  • Apps de sucesso têm em média 3 a 5 telas principais acessíveis com no máximo 2 toques a partir da tela inicial.
  • O Bottom Navigation Bar (barra inferior de navegação) é o padrão mais usado em apps Android modernos segundo o Material Design.
  • Listas dinâmicas com lazy loading melhoram o desempenho ao carregar apenas os dados visíveis na tela.

Fonte: Google Material Design Guidelines; Nielsen Norman Group.

7.2 Criando o Diagrama de Navegação

O diagrama de navegação mapeia todas as telas do app e as conexões entre elas. É o equivalente a um mapa do seu app — antes de construir, você precisa saber para onde cada botão leva. A Figura 7.1 ilustra um exemplo de diagrama de navegação para o app FinanCampus com suas principais telas e fluxos.

Definição

User Flow (Fluxo do Usuário) é um conceito de UX, não uma ferramenta. Ele representa visualmente os caminhos que um usuário pode percorrer dentro do app para completar uma tarefa — mostrando todas as telas e as transições entre elas, incluindo decisões (ex.: logado / não logado). O Diagrama de Navegação é o documento que representa esse User Flow graficamente.

Ferramentas para criar o Diagrama de Navegação

Opção 1 — Whimsical (whimsical.com): ferramenta online, gratuita e criada especificamente para flowcharts e User Flows. Basta criar uma conta, escolher o template Flowchart e começar a adicionar retângulos (telas) e setas (navegações). Não requer instalação e funciona direto no navegador. Recomendada por ser a mais simples e intuitiva para esta tarefa.

Opção 2 — Balsamiq (balsamiq.com): embora seja uma ferramenta projetada principalmente para wireframes de telas (como utilizado na Semana 3), o Balsamiq possui formas básicas e conectores que permitem criar um diagrama de navegação simples. Se você já tem familiaridade com ele, pode aproveitá-lo aqui — apenas saiba que esta não é sua função principal e os recursos de flowchart são mais limitados do que no Whimsical.

Elementos do Diagrama de Navegação

  • Retângulo: representa uma tela (página).
  • Seta: representa a navegação entre telas.
  • Losango: representa uma decisão (ex: usuário está logado?).
  • Rótulo na seta: descreve a ação que aciona a navegação.

Padrões de Navegação no FlutterFlow

7.3 Listas Dinâmicas no FlutterFlow

Uma lista dinâmica exibe dados vindos do banco em um componente de lista (ListView, GridView). No FlutterFlow, você configura a fonte de dados da lista via Backend Query e mapeia os campos para os elementos visuais.

Criando uma Lista Dinâmica

  1. Adicione um ListView ou GridView à tela.
  2. Configure o Backend Query: selecione a tabela e os filtros desejados.
  3. Crie um Card componente para representar cada item da lista.
  4. Mapeie os campos do banco para os textos e imagens do Card.
  5. Configure a ação de clique para navegar ao detalhe do item.

Filtros e Ordenação

No Query Builder do FlutterFlow, você pode adicionar:

  • Filtros (where): exibir apenas registros do usuário logado (usuario_id == currentUserUid).
  • Ordenação (order): ordenar por data, valor ou nome.
  • Paginação (limit): carregar apenas os primeiros N registros para melhorar o desempenho.

A Figura 7.3 representa visualmente o fluxo completo de um ListView com Backend Query: os dados saem da coleção do Firestore, passam pelo filtro configurado no Query Builder e chegam até os widgets Card da tela, com cada campo mapeado via Set from Variable.

Foreign Keys já carregadas pelo Get Schema

Quando o FlutterFlow importou o schema do Firebase via a configuração de coleções na Semana 6, ele também reconheceu as Foreign Keys definidas entre as tabelas. Isso significa que, ao configurar o Backend Query do ListView de Transações, você pode acessar diretamente os campos da categoria relacionada via a Foreign Key. Exemplos:

  • transacao.categoria.nome — exibe o nome da categoria no card do item
  • transacao.categoria.cor — usa a cor da categoria para colorir o ícone do card, mantendo coerência visual com o banco.

Esse acesso via Document Reference é possível porque o Firestore retorna os dados já com os relacionamentos resolvidos quando configurado corretamente na query. Explore esses campos ao mapear os widgets do Card para enriquecer a listagem com dados reais relacionados.
Importante

Sempre filtre os dados pelo usuário logado em listas que exibem informações pessoais. Use o usuario_uid no filtro do Firestore ou currentUserUid no FlutterFlow para garantir que cada usuário veja apenas seus próprios dados.

7.4 Validando com o Storyboard do FlutterFlow

O FlutterFlow possui uma ferramenta chamada Storyboard que oferece uma visão panorâmica de todas as telas do app e suas conexões de navegação. É uma excelente forma de validar visualmente se o diagrama de navegação que você planejou está implementado corretamente.

O Storyboard é somente visualização

O Storyboard do FlutterFlow não permite criar navegações — ele apenas mostra as conexões já definidas nas propriedades de cada página (Actions → Navigate To). Pense nele como uma "planta baixa" do app: mostra o que já existe, mas não constrói nada novo.

Como usar o Storyboard

  1. No FlutterFlow, clique no ícone de Storyboard no menu lateral (ícone de mapa/grid).
  2. Todas as telas do app aparecerão com linhas conectando as navegações:
    • Linha sólida: Navigate To / Navigate & Remove All
    • Linha pontilhada: Bottom Sheet / Dialog
  3. Clique em uma tela para destacar todas as rotas de entrada e saída daquela página.
  4. Compare com o diagrama de navegação que você desenhou — se existem telas desconectadas ou fluxos faltando, é sinal de que alguma Action de navegação não foi configurada.

7.5 Bônus: Publicação Web Rápida

Quer ver seu app funcionando online e compartilhar com amigos antes da publicação final na Play Store? O FlutterFlow permite gerar uma URL pública em poucos minutos:

  1. Acesse Settings → Platforms e habilite Web.
  2. Clique em Web Publishing.
  3. Defina uma URL personalizada para o seu app.
  4. Clique em Publish e aguarde cerca de 5 minutos.
Isso não substitui a Play Store

A publicação web é ótima para testes e demonstrações, mas o app completo será publicado na Google Play Store nas Semanas 12–14. A versão web é um bônus motivacional — aproveite para compartilhar com o professor e colegas!

📱 Estudo de Caso — Semana 7

Mapa de telas — FinanCampus (11 telas):

  • Autenticação: Splash | Login | Cadastro | Recuperar Senha
  • Principal (Bottom Nav — 4 abas): Home/Dashboard | Lista de Transações | Nova Transação | Configurações
  • Detalhes/Edição: Detalhe da Transação | Editar Transação

Diagrama de navegação criado no Whimsical. Fluxo principal: Splash → [usuário logado?] → sim: Home / não: Login → Home. A partir do Home, o usuário acessa as 4 abas pelo Bottom Navigation Bar. O botão flutuante (+) no Home navega diretamente para Nova Transação. Toque em um item da lista navega para Detalhe da Transação, que dá acesso a Editar Transação pelo botão de edição.

Lista dinâmica implementada na tela "Lista de Transações": widget ListView com Backend Query no Firestore, filtrado por usuario_uid = currentUserUid, ordenado por data DESC, limite de 50 registros. Cada item exibe um widget Card com Row contendo: ícone da categoria, descrição, valor e data — todos vinculados via Set from Variable ao resultado da query.
